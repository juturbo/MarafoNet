\section{Architecture}\label{sec:arch}

\begin{frame}
\frametitle{Architectural style}
The system merges the \textbf{data-centric} and \textbf{event-based} architectures. In this way the system has:
\begin{itemize}
    \item \textbf{decoupled communications}: user interactions are asynchronous
    \item \textbf{transparency}: components don't know each other's existence, coordination is managed
        throught the dataspace. Communications and events are delivered throught the dataspace.
    \item \textbf{single source of truth}: the dataspace serves as single source of truths, making the system consistent
\end{itemize}

The server's architecture is divided in stateless microservices, this enables for scalability and
easier lifecycle management. 

\end{frame}

\begin{frame}
\frametitle{Architectural choices}
The stateless microservices architecture solves issues from the shared dataspace architectural style, mainly:
\begin{itemize}
    \item {\color{ForestGreen}\textbf{No single point of failure}}: statelessness allows for replication, avoiding a
        single point of failure
    \item {\color{ForestGreen}\textbf{Scalability}}: multiple stateless microservices can be scaled without data races
    \item {\color{ForestGreen}\textbf{Modularity}}: dividing in microservices makes debugging and lifecycle management easier 
\end{itemize}
Other issues still remain:
\begin{itemize}
    \item {\color{Bittersweet}\textbf{Performance overhead and concurrency}}: multiple services accessing the same 
        dataspace increase overhead and slows down perfomance, making it harder to maintain a consistent state
\end{itemize}

\section{Principles and components}\label{sec:principles}

\begin{frame}
\frametitle{Principles}
In real-world systems network partitions happen often so choosing the P (\textbf{\color{ForestGreen}Partition Tolerance}) of the CAP 
thorem is fundamental.

Having to choose one between \textbf{Availability} and \textbf{Consistency}, we chose to have \textbf{\color{ForestGreen}consistent game data} at the cost
of making the service unavailable in case of a fault.

Availability can be {\color{Bittersweet}eventually built in the infrastructure} through \textit{restart policies, replication and load balancing}.
\end{frame}

\begin{frame}
\frametitle{System's features}
Our design implements features from distributed systems:
\begin{itemize}
    \item \textbf{Consistency and Integrity}: game's data is strongly consistent and integrity is ensured by the use 
        of \texttt{etcd} together with \texttt{Kubernetes}
    \item \textbf{Availability}: even thought it's not a priority, Kubernetes employs restart policies
        and replication mechanisms to maximise Availability
    \item \textbf{Maintainability}: the stateless microservices architecture distributed on Kubernetes makes 
        updates rollout and debugging easier
\end{itemize}
\end{frame}

\section{Storage}\label{sec:storage}

\begin{frame}
    \frametitle{Storage}
    Needing \textbf{strong consistency}, we needed a technology implementing a consensus algorithm
    such as Paxos or Raft.

    These algorithms enable consistency in data by managing \textbf{consensus} between nodes, ensuring they all store the same data.
    
    This feature is used to guarantee fault-tolerant consensus and replication.
\end{frame}

\begin{frame}
\frametitle{Storage choice: \texttt{etcd}}
Etcd implements Raft, ensuring that a cluster with $n=2f+1$ nodes can survive $f$ failures.

The Go etcd library also offers publish-subscribe notifications on data changes and leases.

\begin{block}{Remark}
Etcd \alert{is not protected by Byzantine Failures}: in Raft every node \textbf{assumes that all the other
nodes are honest}, so a malicious behavior may result in the breaking of the vote.
\end{block}
Byzantine failures usually occour on components exposed to an external network, vulnerable to malicious attacks.
The etcd cluster is inside a private and trusted network, behind the backend so it's not vulernable.
\end{frame}

\begin{frame}
\frametitle{Storage alternatives}
Even thought geographical distribution is not a concern, etcd lacks scalability and concurrency features on large
amounts of data.

Another choice could have been CockroadDB. This choice would add complexity but enable for better scalability:
\begin{itemize}
    \item {\color{ForestGreen}\textbf{multi-Raft}}: multiple leaders can write concurrently to different \textbf{data shards}
    \item {\color{ForestGreen}\textbf{linear scalability}}: adding nodes actually boosts performance on data accessing
    \item {\color{ForestGreen}\textbf{CDC (Change Data Capture)}}: like etcd watchers, can be connected to external pub-sub brokers
    \item {\color{Bittersweet}\textbf{relational database}}: more complex design and access than simple key-value storage
\end{itemize}
\end{frame}
